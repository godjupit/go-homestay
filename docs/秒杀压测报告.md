# 秒杀压测方法与报告

这份报告不预填虚假的 QPS。每次展示前应在明确的机器和参数上重新执行，并把原始结果随报告保存。

## 测试目标

秒杀压测不仅观察吞吐量，还必须验证：

1. 同一用户重复请求只扣减一次库存。
2. 多用户并发时成功数不超过库存。
3. Redis 库存、MySQL 售出量、秒杀订单数最终一致。
4. 异步创建订单失败时只补偿一次。
5. 所有已接受预约最终进入 `success` 或 `failed`，不能永久停留在 `pending`。

## 执行步骤

仓库内置的可重复测试使用隔离活动 `9001`：

```bash
make benchmark-seckill
```

1. 使用独立活动 ID，避免修改演示活动和开发订单。
2. 先使用同一 Token 并发请求，验证一人一单。
3. 再使用不同用户 Token 竞争剩余库存。
4. 轮询预约结果，等待异步 Worker 完成。
5. 同时查询 Redis 和 MySQL，记录最终不变量。
6. 采集请求延迟的 P50、P95、P99 和错误分类。

推荐将原始输出保存到未提交的本地目录 `interview-practice/results/`，报告只提交脱敏后的汇总。

## 2026-07-16 并发正确性基线

这次本地 Docker Compose 运行只用于验证正确性，不作为性能容量数据：同一用户并发请求 10 次只产生 1 个预约；8 个用户竞争剩余 4 份库存，4 个成功、4 个售罄；最终 Redis 库存为 0，MySQL `sold_count`、订单数和唯一用户数均为 5，没有超卖。延迟分位数仍需在固定机器配置下另行采集。

## 面试说明

重点不是追求无法复现的高 QPS，而是解释 Lua 原子预扣、一人一单、Redis Stream 消费、MySQL 唯一约束、失败补偿和最终一致性分别解决了什么问题。单机 Docker Compose 的结果不能直接等价为生产容量。
